# アスミツ履歴書 ロードマップ

---

## 前提

このロードマップは、コンセプト（concept.md）で定義した内容を**いつ・何を・どの順番で実行するか**を示すもの。

起点: 2026年4月

---

## 全体像

```
Week 1（今週）:     MVP開発（2〜3日）
Week 2-4（1ヶ月目）: AI品質テスト + SNS訴求準備 + 既存リストテスト
Month 2（2ヶ月目）:  SNS訴求本格開始 + 送客検証
Month 3（3ヶ月目）:  データを見て改善 + 続けるか判断
```

## Week 1: MVP開発（2〜3日）

### 作るもの

| 機能 | 内容 |
|------|------|
| 履歴書フォーム | JIS規格の穴埋め式。スマホ対応 |
| AI志望動機生成 | 職種・経歴を入力→AIが3パターン提案 |
| AI自己PR生成 | 同上 |
| PDF出力 | 完成した履歴書をPDFでダウンロード |
| 送客ポップアップ | 履歴書完成後に「条件が良くなる求人サンプル」→エージェント送客 |

### 作らないもの（後回し）

| 機能 | 理由 |
|------|------|
| アカウント機能 | 最初は保存なし。作ったらPDF出力して終わり。検証には不要 |
| コンビニ印刷 | 開発に時間がかかる。PDFをコンビニで印刷する方法を案内すれば代替可能 |
| データ保存 | アカウント機能と一緒に後回し |
| スマホアプリ | Webで十分。アプリは後から |
| 凝ったデザイン | 最低限使えるUIでOK。デザインは後から改善 |
| 分析ダッシュボード | 最初はGoogleアナリティクス等で代替 |

### リスクと対策

| リスク | 影響 | 対策 |
|--------|------|------|
| PDF出力でJIS規格の履歴書フォーマットが崩れる | 印刷した時に使い物にならない | 既存のPDFテンプレートやライブラリを活用。ここの調整が最も時間かかる可能性あり |
| AIの生成品質が低い | 試したユーザーが「微妙」と感じて離脱 | 開発中にプロンプト調整。複数モデルで比較。「自分で書くよりマシ」レベルならOK |
| スマホでの入力体験が悪い | フォーム入力が面倒で途中離脱 | 入力項目を最小限に絞る。必須項目以外は後から追加できる設計 |

## Week 2-4（1ヶ月目）: AI品質テスト + SNS訴求準備 + 既存リストテスト

### AI品質テスト

MVPが動く状態になったら、実際の職種パターンでAIの出力をテストする。

**テストすべき職種パターン（この層に多い職種）：**
- 工場勤務（製造・検品・ライン作業）
- 販売職（アパレル・携帯ショップ）
- 飲食（ホール・キッチン・店長）
- 介護（介護士・ヘルパー）
- 事務（一般事務・データ入力）
- コールセンター
- ドライバー

**確認ポイント：**
- 志望動機が「それっぽい」レベルで出るか
- 自己PRが経歴に基づいた内容になっているか
- 敬語が不自然でないか
- 入力情報が少ない場合（経歴1社のみ等）でもまともに出るか
- 全職種で同じような文章にならないか（テンプレ感がないか）

### SNS訴求の準備

サービスが動く状態になったら、SNS投稿の準備を始める。

**やること：**
- 実際にサービスを使って履歴書を作る過程を画面収録
- 「正直、履歴書の書き方わからんかった→AIが全部作ってくれた→無料」のストーリーで動画を作る
- 投稿前に数パターンの訴求を準備しておく
  - 層A向け：「AIが書いてくれる」系
  - 層B向け：「通過率上がった」系
  - 共通：「2週間で内定」系

### 既存LINEリスト（33,000人）へのテスト配信

**SNS投稿より先に、既存リストでテストする。**

理由：
- SNSは投稿してみないと反応がわからないが、LINEリストなら確実に届く
- 33,000人のうち一部でもサービスを使ってくれれば、初期データが取れる
- 送客の導線が機能するかを、SNS本格開始前に検証できる

**配信内容（案）：**
```
「履歴書の志望動機と自己PR、AIが無料で作ってくれるサービスができました」
「アスミツ履歴書で検索してみてください」
```

**計測する指標：**

| 指標 | 何がわかるか |
|------|------------|
| 開封率 | そもそも興味があるか |
| クリック率（サービスへの遷移） | 「使ってみたい」と思う人の割合 |
| 履歴書作成の完了率 | 最後まで使える品質か |
| AI機能の利用率 | AI志望動機・自己PRを使う人の割合 |
| 送客のタップ率 | ポップアップで「気になる」を押す割合 |

### 1ヶ月目のリスクと対策

| リスク | 影響 | 対策 |
|--------|------|------|
| 既存リストの反応が薄い | 初期データが取れない | 配信文言を変えて再配信。「適職診断の結果を活かして履歴書を作ろう」等の切り口も試す |
| 送客ポップアップが押されない | マネタイズの成立が見えない | ポップアップの文言・タイミング・デザインを変えて検証。送客先のエージェントを変える |
| AIの品質にクレームが来る | サービスの信頼性が下がる | テスト段階で品質を担保。問題があればモデルを変更（コストは安いので高品質モデルでも問題ない） |

## Month 2: SNS訴求本格開始 + 送客検証

### SNS訴求の本格開始

既存リストでの初期検証を踏まえて、SNSでの投稿を本格開始する。

**投稿計画：**
- TikTok/Instagram/Threadsに週3〜5本投稿
- 最初の2週間で複数の訴求パターンを試す
  - 層A向け：「履歴書の書き方わからない人、AIが全部作ってくれるよ」
  - 層B向け：「書類で落ちまくってた→これ使ったら通った」
  - 共通：「このサービス使って2週間で内定出た」
- 反応が良い訴求を特定して、そこに集中する

**適職診断のSNSとの関係：**
- 別アカウントで運用するか、同じアカウントで投稿するかを決める
- 別アカウントの方がターゲットが明確になるが、ゼロからフォロワーを集める必要がある
- 同じアカウントなら既存フォロワーにリーチできるが、適職診断との訴求が混在する

### 送客の本格検証

SNSから来たユーザーで、送客の導線が機能するかを検証する。

**既存リストのユーザーとSNSからのユーザーでは行動が違う可能性がある：**

| | 既存リスト経由 | SNS経由 |
|--|-------------|---------|
| サービスへの信頼度 | ある程度ある（適職診断を使ったことがある） | ゼロ（初めて知った） |
| 温度感 | バラバラ（潜在層も混在） | 訴求に反応した人だけが来る |
| 送客の受け入れ | わかりにくい | 訴求内容次第で変わる |

**だからSNS経由のデータを別で計測する必要がある。**

### 2ヶ月目のリスクと対策

| リスク | 影響 | 対策 |
|--------|------|------|
| SNSの訴求がバズらない | ユーザーが来ない | 訴求パターンを高速で回す。適職診断でバズった手法を応用。最初の2週間で10パターン以上試す |
| 「怪しいサービス」と思われる | SNSでネガティブな反応 | 段階的情報取得（最初は個人情報なし）で信頼を獲得。プライバシーポリシーを明示 |
| SNS→Google検索の導線で離脱 | 検索してもサービスが見つからない | 「アスミツ履歴書」の指名検索でSEO1位を確保。リスティング広告で補完も検討 |
| 送客の転換率が適職診断と変わらない | 「顕在層だから高い」という仮説が崩れる | 送客のタイミング・文言・送客先を変えて検証。仮説が間違っていた場合はマネタイズ方法を再検討 |

## Month 3: データを見て改善 + 続けるか判断

### 2ヶ月分のデータを分析

この時点で、以下のデータが揃っているはず：

**集客データ：**
- SNSの投稿ごとの反応（再生数、いいね、コメント、保存）
- SNS→Google検索→サービスへの流入数
- どの訴求パターンが最も人を動かしたか

**サービス利用データ：**
- サービスに来た人の数
- 履歴書作成の完了率
- AI志望動機・自己PRの利用率
- どこで離脱しているか

**マネタイズデータ：**
- 送客ポップアップのタップ率
- エージェントへの送客数
- 送客→面談→成約の転換率
- 月間の送客収益

### 改善

データに基づいて改善する。

**よくある改善パターン（想定）：**

| データ | 考えられる問題 | 改善策 |
|--------|-------------|--------|
| SNSの再生数は多いがサービスへの流入が少ない | Google検索の導線で離脱している | CTAの文言変更、リスティング広告の併用 |
| サービスに来るが履歴書を最後まで作らない | 入力項目が多すぎる or UIが使いにくい | フォームの簡略化、入力ステップの最適化 |
| 履歴書は作るがAI機能を使わない | AI機能の存在に気づいていない or 使い方がわからない | UIでAI機能をもっと目立たせる |
| AI機能は使うが送客ポップアップが押されない | ポップアップのタイミングか内容が悪い | 文言・デザイン・タイミングを変更 |
| 送客はするが面談に至らない | 送客先のエージェントがこの層に合っていない | 送客先の変更。高卒/専門卒向けエージェント（ハタラクティブ等）に切り替え |

### 3ヶ月後の判断

**最大の判断ポイント：「このサービスを続けるか」**

| 判断基準 | 続ける | 方向転換を検討 |
|---------|--------|-------------|
| SNSで人が来るか | 月間数百〜千人が来ている | ほとんど来ない |
| 送客が成立するか | 月に数件以上のエージェント送客が発生 | 送客ゼロ or タップすらされない |
| ユニットエコノミクス | 送客1件あたりの収益 > 集客コスト | 収益が出ない構造 |

**「続ける」の場合：** Month 4以降でスケールフェーズに入る（SNS投稿増、サービス改善、SEO開始）

**「方向転換」の場合：** 何が悪かったかを分析して：
- 訴求が悪い → 訴求を変えて再挑戦
- サービスの使い勝手が悪い → UI/UXを改善して再挑戦
- そもそもこの層に需要がない → コンセプト自体を再検討

**3ヶ月で「完璧な結果」を求めない。** 「送客が数件でも発生した」「SNSで反応がある訴求パターンが見つかった」レベルで十分。それがあれば、4ヶ月目以降で改善・スケールできる。
